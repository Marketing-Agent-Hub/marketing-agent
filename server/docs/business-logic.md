# Business Logic

## Core Business Rules

### Multi-Tenancy Model
- The top-level entity is a **Workspace**, which is a logical container for a team.
- Each Workspace has **Brands** — a Brand represents one client or product being marketed.
- Users belong to Workspaces via **WorkspaceMember** with a role: `OWNER > ADMIN > EDITOR > VIEWER`.
- All brand-level data (strategies, content, sources) is isolated by `brandId`.
- **Rule**: A user can only access a Brand if they are a member of its parent Workspace.

### Content Intelligence: Item Lifecycle

Items (news articles) flow through a strictly ordered state machine:

```
NEW → EXTRACTED → READY_FOR_AI → AI_STAGE_A_DONE → AI_STAGE_B_DONE
                ↘                    ↘
            FILTERED_OUT         FILTERED_OUT
                                (if Stage A rejects)
```

- **NEW**: Item was fetched from a feed but not yet processed.
- **EXTRACTED**: Full article content fetched from the source URL, stored in `Article` table.
- **READY_FOR_AI**: Passed keyword filter and optionally AI embedding similarity filter.
- **AI_STAGE_A_DONE**: Classified, tagged, scored by AI. `isAllowed` field determines if it proceeds.
- **AI_STAGE_B_DONE**: A full Vietnamese Facebook post has been generated.
- **FILTERED_OUT**: Item was rejected at any stage. `filterReason` stores why.

**Rule**: Each stage service checks and enforces the expected status before processing. A job will silently skip items not in the expected state.

### Deduplication
- Items are uniquely identified by `(brandId, sourceId, link)` — a unique constraint in Postgres.
- When saving items, a `P2002` (unique constraint violation) is caught and counted as `existing`, not as an error.
- Content deduplication for AI Stage B uses `contentHash` — if an identical article was already processed, the cached `fullArticle` is reused without another AI call.

### Source Trust Score
- Each `Source` has a `trustScore` (0–100).
- Sources below a threshold are not used for filtering comparisons.
- `denyKeywords` per source are applied in addition to global keyword filters.

### Filter Engine Logic (for Marketing Content)
Two modes, set per-brand via `FilterProfile.mode`:

**`PASS_THROUGH`**: All extracted items are marked `READY_FOR_AI` without scoring.  
**`AI_EMBEDDING`**: Items are embedded using an AI model (e.g., `text-embedding-3-small`) and compared against a `vectorProfile` (a pre-computed vector of the brand's ideal content). If cosine similarity < `similarityThreshold` (default: 0.7), the item is filtered out.

### Marketing Strategy Rules
- A Brand can only have **one ACTIVE strategy** at a time.
- Activating a new strategy automatically sets all current ACTIVE strategies to `SUPERSEDED` (via a Prisma transaction).
- `StrategySlot` represents a single planned post — a position in time with a platform, pillar, and funnel stage.
- Slots progress: `PLANNED → BRIEF_READY → DRAFT_READY → APPROVED/SKIPPED`
- The daily content job only processes slots with status `PLANNED` and `scheduledFor <= cutoff`.

### Content Approval Workflow
- Drafts generated by AI start in `IN_REVIEW` status.
- An `EDITOR` or above can approve (`APPROVED`) or reject (`REJECTED`).
- Only `APPROVED` drafts can be scheduled for publishing.
- After approval, an editor specifies a `scheduledFor` datetime to create a `PublishJob`.
- `REJECTED` and `IN_REVIEW` drafts are archived when a regeneration is requested (they cannot be deleted, only archived).

### Publishing
- The `PublishSchedulerJob` runs every 5 minutes and queries for `PublishJob` records with `status: SCHEDULED` and `scheduledFor <= now`.
- Currently all publishing uses a **`StubConnector`** — no real social platform API is called; a simulated response is returned.
- After publishing, a `PublishedPost` record is created. Analytics can be collected via `PerformanceSnapshot`.

## Key Workflows (Step-by-Step)

### Onboarding a New Brand

1. Admin/Editor creates a new `Brand` under a Workspace via `POST /api/workspaces/:id/brands`.
2. Frontend calls `POST /api/brands/:brandId/onboarding/sessions` to create an `OnboardingSession`.
3. Q&A dialog proceeds — each turn calls `POST /api/brands/:brandId/onboarding/sessions/:sessionId/messages` with `{role, content}`.
4. When onboarding is complete, `POST .../:sessionId/complete` is called.
5. This triggers an async `OnboardingAnalysisJob` (via `setImmediate`) which calls the AI with the full transcript and creates a `BrandProfile` — extracting `targetAudience`, `valueProps`, `toneGuidelines`, `businessGoals`, and `messagingAngles`.
6. The brand is now ready for strategy generation.

> **Edge case**: If the transcript is empty at completion time, a `VALIDATION_ERROR` (422) is thrown. If the session already has status `COMPLETED`, a `INVALID_STATE_TRANSITION` (422) is thrown.

### Generating a Content Strategy

1. Requires `BrandProfile` to exist (422 if missing).
2. `POST /api/brands/:brandId/strategies/generate` with optional `{durationDays, postsPerWeek, startDate, channels}`.
3. Service pulls brand profile + pillars, calls AI (`strategy-generation` workflow) to generate a `StrategyPlan` with weekly themes and timed slots.
4. AI output is validated with Zod schema before being persisted.
5. The plan is stored in a **single Prisma transaction**: `StrategyPlan` created first, then `StrategySlot` records bulk-created via `createMany`.
6. Strategy starts in `DRAFT` status and must be explicitly activated.

### Daily Content Generation

1. Runs at 06:00 daily per brand (triggered by cron, or manually via `POST /api/brands/:brandId/content/generate-daily`).
2. Refreshes `TrendSignal` records from recent AI-processed items.
3. Runs `matchBrandToRecentSignals` — embeds the brand's key topics and compares cosine similarity against recent trend signals. Matches above 0.6 threshold are saved as `BrandTrendMatch`.
4. Fetches all `PLANNED` slots with `scheduledFor <= today + daysAhead`.
5. For each slot:
   - AI generates a `ContentBrief` (title, objective, key angle, CTA).
   - AI generates a `ContentDraft` (hook, body, CTA, hashtags) using the brief as context.
   - If a relevant `TrendSignal` was matched, it is injected as context into both prompts.
   - Both are stored; the slot status advances to `DRAFT_READY`.
6. Drafts start in `IN_REVIEW` status awaiting human review.

## Edge Cases Handled in Code

| Scenario | Handling |
|---|---|
| Duplicate news item URL | Caught by unique constraint `(brandId, sourceId, link)` via `P2002` error code; counted as `existing`, not thrown |
| OpenRouter credit exhausted (HTTP 402) | `OpenRouterCreditError` causes immediate batch abort — no further items are attempted |
| OpenRouter service overloaded (HTTP 529) | `OpenRouterOverloadedError` triggers exponential backoff (1s, 2s, 4s) up to 3 retries |
| Article content not extractable | Readability returns null; item is still marked `EXTRACTED` with `snippet` used as fallback content |
| Stage A AI rejects item | Item gets `filterReason` set and status `FILTERED_OUT`; Stage B skips it |
| Empty AI response | Throws `Error('No response from OpenAI')` — item stays in current status for retry |
| Invalid AI JSON output | `JSON.parse` or Zod validation fails → error is caught, item fails gracefully |
| Strategy not in DRAFT on activate | Throws 422 `INVALID_STATE_TRANSITION` |
| Brand has no profile on strategy generate | Throws 422 `PRECONDITION_FAILED` |
| Redirect URLs (Google News, bit.ly) | `resolveActualUrl()` follows HEAD request redirects up to 5 hops before extracting |
| Concurrent discovery runs | `isRunning` boolean guard prevents overlapping execution |
| AI Stage B content hash cache | If same article was already processed (same `contentHash`), reuses existing `fullArticle` — no duplicate AI calls |
